## Objective

实现 issues 事件处理程序，能够解析 GitHub webhook 负载，提取 issue 数据（标题、作者、动作、URL、标签），并将事件格式化为可读的通知消息。处理程序应支持 opened、closed、reopened 动作，并为 deleted 动作提供特殊处理。

**Purpose:** 作为第一个实际事件处理程序，为后续的 PR、comment 和 release 处理程序提供实现模式。issues 事件是最常见的 GitHub 事件之一，处理完成后用户可以实时接收仓库 issue 的状态变化通知。

**Output:** 完整的 issues 事件处理程序，包含输入验证、动作过滤、标签格式化和消息构建功能，以及全面的单元测试覆盖。

### Task 1: 创建 issues 事件处理程序基础结构和输入验证

**Files:** `src/lib/handlers/issues.js`, `src/lib/handlers/index.js`

**Action:**
创建 `src/lib/handlers/issues.js` 文件，实现 `handleIssues(payload)` 函数，包含：
1. 输入验证：检查 payload 是否为对象（遵循现有占位符模式）
2. Action 字段验证：如果 payload.action 缺失，抛出错误（per D-07）

- number: 如果为 null，跳过 URL 生成（per D-15）
- sender: 如果为 null，使用占位符 `@unknown`（per D-16）
- labels: 提取 name 和 color 字段

### Task 2: 实现消息格式化逻辑

**Files:** `src/lib/handlers/issues.js`

**Action:**
在 `handleIssues` 函数中添加消息格式化逻辑，创建私有辅助函数 `formatIssueMessage(issueData, action)`：

1. **动作标签格式化**（per D-09, D-10, D-11, D-12）:
   - opened: `🔓 Issue Opened by @{username}`
   - closed: `🔒 Issue Closed by @{username}`
   - reopened: `♻️ Issue Reopened by @{username}`
   - deleted: `🗑️ Issue Deleted by @{username}`（特殊格式，不包含标签行）

2. **标签行格式化**（per D-02）:
   - 遍历 labels 数组，为每个标签添加对应颜色的圆点 emoji
   - 颜色映射规则：
     - red → 🔴
     - green → 🟢
     - blue → 🔵
     - yellow → 🟡
     - purple → 🟣
     - orange → 🟠
     - pink -> 🩷
     - gray → ⚪
     - 其他 → ⚫
   - 标签格式：`{emoji} {name}`，多个标签用空格分隔
   - deleted 动作跳过标签行（per D-06）

3. **issue 信息行格式化**（per D-04）:
   - 格式：`#{number}: {title}`
   - 长标题不截断，允许自动换行

4. **URL 行格式化**:
   - 格式：`🔗 {html_url}`
   - 仅当 number 不为 null 时包含此行

5. **最终消息组装**:
   - 使用标准两行格式（per D-01）
   - 动作标签 + 标签行（如果是非 deleted 动作且有标签）
   - issue 信息行
   - URL 行（如果有）

6. **返回值**:
   - processed: true（对于支持的动作）
   - message: 格式化的消息字符串
   - event: 'issues'
   - data: 包含 action、number、title、sender、labels 的对象

## Threat Model

### Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub → Webhook Server | Untrusted network input, signature verified upstream |
| Handler → Formatter | Internal boundary, trusted data after validation |

### STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Spoofing | payload.sender | accept | GitHub signature verification in Phase 2 prevents spoofing. No additional mitigation needed. |
| T-04-02 | Tampering | payload.action | mitigate | Validate action field exists and is in supported set. Throw error if missing (D-07). |
| T-04-03 | Tampering | payload.issue | mitigate | Validate issue object exists (D-13). Use placeholders for missing optional fields (D-14, D-15, D-16). |
| T-04-04 | Information Disclosure | Log output | mitigate | Never log full payload or sensitive data. Use app.log.warn with minimal context for unsupported actions (D-05). |
| T-04-05 | Denial of Service | Malformed payload | mitigate | Input validation at handler entry prevents crashes. Return processed: false for unsupported actions rather than throwing. |
| T-04-06 | Escalation | N/A | N/A | N/A - No permission escalation in this phase (read-only handler) |
