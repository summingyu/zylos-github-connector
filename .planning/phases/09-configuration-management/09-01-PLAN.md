<task type="auto">
  <name>Task 1: 实现深度默认值合并和配置验证</name>
  <files>src/lib/config.js</files>
  <action>
    重构 src/lib/config.js 以实现以下功能：

    1. **深度默认值合并**：
       - 创建 mergeDefaults 函数，递归合并嵌套对象
       - 合并策略：用户配置覆盖默认值，但嵌套对象（logging、commBridge）需要深度合并
       - 示例：{ ...defaults.logging, ...userConfig.logging }

    2. **配置验证**：
       - 创建 validateConfig 函数，验证必需字段和值范围
       - 验证规则：
         * webhookSecret: 字符串，长度 >= 16（或明确标记为测试环境）
         * port: 数字，范围 1-65535
         * logging.level: 枚举值 ['error', 'warn', 'info', 'debug']
         * enabled: 布尔值
         * commBridge.enabled: 布尔值
       - 验证失败抛出 Error with 描述性消息

    3. **敏感字段保护**：
       - 创建 sanitizeForLogging 函数，移除敏感字段
       - 移除 webhookSecret 字段，替换为 '[REDACTED]'
       - 在 loadConfig 中记录配置时使用此函数

    4. **重构 loadConfig 函数**：
       - 使用 fs.promises.readFile 而非 fs.readFileSync（异步）
       - 捕获 JSON.parse 错误，提供详细错误信息（行号、列号）
       - 处理 ENOENT 错误（文件不存在），返回默认配置
       - 应用环境变量覆盖（GITHUB_WEBHOOK_SECRET）
       - 调用 mergeDefaults 和 validateConfig
       - 记录配置加载（使用 sanitizeForLogging）

    5. **更新 DEFAULT_CONFIG**：
       - 添加 maxPayloadSize: '10mb'
       - 添加 settings: {} 空对象用于扩展
       - 确保 logging 和 commBridge 有完整默认值

    **不要**：
    - 不要在此任务中实现热重载（下一个计划）
    - 不要修改配置文件路径
    - 不要移除现有的 getConfig、saveConfig、watchConfig、stopWatching 函数签名
  </action>

  </verify>
  <done>

    - validateConfig 函数验证所有必需字段和值范围
    - sanitizeForLogging 函数移除 webhookSecret
    - loadConfig 使用异步 API、错误处理、默认值合并、验证
    - DEFAULT_CONFIG 包含所有必需字段的默认值
